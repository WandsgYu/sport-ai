# Sport-LM：业务办理 Agent 工程复盘

> 本文只描述脱敏后的工程方法和统计口径，不包含组织名称、真实用户、私有 SOP、生产字段、内部地址或凭据。

## 业务问题

原流程需要用户理解规则、确认自身状态，并在多个选项之间完成查询或变更。普通表单能承载固定字段，却很难处理用户用自然语言表达的不完整需求、追问、修改和异常情况。项目目标不是做一个“什么都能聊”的机器人，而是把自然语言入口约束在一组可验证的业务流程中。

## 为什么使用 Agent

Agent 负责把对话转成受控操作：

1. 识别用户当前要处理的场景；
2. 只加载该场景所需的 SOP 片段；
3. 收集缺少的参数；
4. 对状态变更要求用户明确确认；
5. 通过结构化工具调用查询或提交；
6. 只有工具返回成功后才向用户声明完成。

企业微信承担消息入口和身份上下文，业务系统仍是最终数据权威。LLM 不直接访问数据库，也不能绕过工具侧权限。

## 生产链路

```text
企业微信消息
  → 身份映射与输入检查
  → 意图 / 场景选择
  → 最小 SOP 上下文
  → LLM 参数收集与 Function Calling
  → 主体级鉴权
  → 业务适配器
  → 结果校验与用户回复
  → 脱敏事件与运行监控
```

## 上线后遇到的失败模式

- 用户只说“帮我办一下”，缺少必要参数；
- 用户在确认前改变选项，需要废弃旧的待确认状态；
- 消息平台重试导致同一确认重复到达；
- 模型在工具失败时仍倾向生成自然语言“成功”；
- SOP 片段过多时，模型混用无关规则；
- 身份映射缺失或过期时，工具调用必须在适配器前被拒绝；
- 日志若直接保存对话，排障便利会与隐私要求冲突。

对应措施包括参数状态机、显式确认、幂等键、工具结果门控、场景优先检索、工具侧主体鉴权，以及只记录白名单事件元数据。

## 结果与统计口径

- **300+ 用户**：上线运行期间由消息渠道去重后的服务对象数量级；
- **1200+ 轮业务对话**：按用户请求与系统响应组成的业务交互轮次汇总；
- **稳定性表述**：上线运行期间，未发现因本系统导致的已知业务数据异常或安全事故。

这些数字用于说明实际使用规模，不代表公开仓库中的合成演示产生了相同流量。

## 如果重新做一次

1. 从第一天就把适配器定义成明确协议，生产、沙箱和 Mock 使用同一契约；
2. 用持久化状态机替代进程内历史，明确每次确认对应的版本；
3. 将幂等键、审计事件和业务结果关联成统一 operation ID；
4. 为场景选择建立离线评测集，而不是只依赖线上观察；
5. 将“模型回答质量”和“业务操作正确性”拆成两套指标；
6. 为失败工具调用设计更明确的补偿和人工接管路径。
